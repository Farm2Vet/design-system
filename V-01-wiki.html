01 / Disease wiki — In context

The list.

The whole landing rendered as one viewport. A topbar names the page and offers the only write-action on the surface; the toolbar carries search, three filter dropdowns, and a sort; the list itself is ten rows that read row-by-row from left to right — codename, version, optional incomplete warning, last-updated stamp, active status. The pager footer carries a counter on the left and a five-control cluster on the right.

02 / Disease wiki — Anatomy

Regions.

Four named regions, top to bottom. Each is a layout slot; the controls inside each slot resolve to canonical components (see section 03).

Topbar
Page title .app-frame__title on the left, primary write-action .btn--cta on the right. Subtitle sits one line below the title for a one-sentence scope reminder.
Toolbar
Five controls on a single row. The search input (default .input at LG with a leading magnifier glyph) owns the elastic column. Three filter .dropdown triggers — Version, Completeness, Status — and one sort .dropdown trigger sit in fixed columns to its right. Below 900px the grid collapses to two columns; below 600px it collapses to one.
List
A stack of .disease-row elements, divided by border-default-subtle hairlines. Each row carries the codename (medium weight), a .badge.is-primary version pill, an optional .badge.is-warning incomplete badge, a mono last-updated stamp, and a leading-dot active-status badge. Row affordances stop at display — drill-in, edit, and deactivate live behind the next view, not this one.
Pager
Counter on the left (Showing X–Y of N diseases, mono eyebrow), five-control cluster on the right: first / previous / page-jump input / next / last. The jump is one composite of three children — Page label, input--sm, of N label — so it reads as a single widget rather than three loose pieces.
03 / Disease wiki — Components in use

Reuse, not approximation.

Every interactive element on the mockup is a canonical component. The page-CSS in page-V-01.css only contributes layout scaffolding — the viewport frame, the row grid, the pager strip. If a control here looks broken against its component page, fix the component page, not the mockup.

Search input
.input (default LG, see III-03) with a page-scoped leading-icon wrapper .app-frame__search.
Filter triggers
Three .dropdown instances with full dropdown__trigger  / dropdown__menu  / dropdown__option markup (see III-04). Selected state uses .is-selected; the resting trigger reads as {Filter} · All.
Sort trigger
Same .dropdown markup as the filters. Value reads Sort · {key} · {direction}.
Version pill
.badge .badge--sm .is-primary — soft style, primary tone. The same tone the system uses for selected / brand state across the rail (see III-09).
Incomplete warning
.badge .badge--sm .is-warning with a .badge__icon slot carrying a triangle glyph. Soft style only — the row is still readable, the badge is a flag, not a stop.
Active-status chip
.badge .badge--sm .is-safe for Active, .badge .badge--sm .is-neutral for Inactive. Both carry a leading .badge__dot so the chip reads as a status, not a label.
New-disease CTA
.btn .btn--cta .btn--md with a .btn__leading plus glyph and a .btn__label. Single primary action on the surface.
Pager arrows
Four .btn .btn--ghost .btn--sm icon buttons, square-capped by the page-scoped .pager__btn width override. Each carries an aria-label so screen-readers hear "First page", "Previous page", etc.
Page-jump input
.input .input--sm with a page-scoped .pager__jump-input width override (5ch, centered, tabular-nums) so it reads as a numeric register, not a free-form text field.
