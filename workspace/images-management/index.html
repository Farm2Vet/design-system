Image Management — wireframe set

Greyscale lofi wireframes for FR-331 (view / update) and FR-332 (create). These are structural — no brand colour, no glass, no real components yet — intended to lock down layout and flow before a hi-fi pass that adopts the Farm2Vet design system.

01 · UC-331

List view

Image-first grid with search, filter chips, sort, per-page selector and a primary Add image action. Each card hovers in edit / delete icons; pagination at the foot.

GET /admin/images
02 · UC-331

Empty / no-results state

Search + filters return zero matches. The toolbar stays put so the user sees why there are no results; the body offers clear-filters and add-image affordances.

GET /admin/images?q=…
03 · UC-331 fail

Unauthorized

Signed-in user without the Knowledge-base editor role. Page chrome remains so they know where they landed; body explains and offers a request-access affordance.

GET /admin/images · 403
04 · UC-331

Edit single image

Two-column layout: preview + replace on the left; Disease, English, Vietnamese and Source fields on the right. Source is split into title + URL so admins never paste raw <a href> again. Sticky save bar at the foot.

GET /admin/images/{id}
05 · UC-332

Add new image

Same two-column shape as edit, but the left column is a drag-drop upload zone with file-picker fallback. Submit is disabled until the file and all required fields are present.

GET /admin/image-input
06 · UC-332 state

Uploading + validation

File selected, preview rendered, upload at 68% with a cancel control. Vietnamese-description field shows an inline validation error. Footer surfaces a status line and keeps Submit disabled.

POST /admin/image-input (mid-action)

Open questions to resolve before hi-fi

  • Should Disease be a typeahead-only dropdown (only known diseases) or allow free entry that creates a new disease record?
  • Do we need a separate multi-source model for one image, or is one source per image enough?
  • The current page shows raw <a href> in the source column — confirm the new model is a split title + URL pair, rendered server-side.
  • Bulk actions on the list view: delete, re-tag disease, export? Not designed here; flag if needed.
  • Should the list support a table toggle for power users, or stay grid-only?
