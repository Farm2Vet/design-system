Section 08 / Components Modal Dialog.

A blocking surface that interrupts the task flow until the user resolves it. One shared shell composes a header, body, and footer, and resolves through component-scoped token slots (--modal-*) into seven conventional shapes: confirmation, alert, form, scrollable, drawer, bottom-sheet, and fullscreen.

Architectural note · scope boundaries

Modal is the blocking-message surface. A non-blocking, in-flow message is an Alert. A transient, auto-dismissing message is a Toast. A contextual surface anchored to a trigger (without a scrim, without focus trap) is a Dropdown or Popover. Modal is reserved for surfaces that must interrupt the underlying view: destructive confirmations, focused tasks, and content that needs a focus boundary to be usable.

Architectural note · one modal at a time

Stacked modals are not part of this system. Opening a second dialog over a first nests two focus traps and creates a context the user cannot reason about. When a flow needs more than one step, sequence the steps inside one dialog (multi-step) or promote the flow to a dedicated route. The single exception is a Critical alert dialog raised from inside another dialog to confirm a destructive action on its own contents — handled by replacing, not layering, the surface.

01 / Modal — Anatomy

A shared shell.

Every modal resolves into the same vertical shell: a backdrop (.modal), a surface (.modal__surface), a header (.modal__header) carrying the title and the dismiss control, a body (.modal__body), and a footer (.modal__footer) for actions. Variants differ only in which slots they expose and how the surface is anchored on the viewport.

Title is the only required slot. A modal without a title cannot be referenced by assistive tech (aria-labelledby resolves to the title element) and has no docking handle for the user's attention. Description, body, and footer are optional — a one-line confirmation can omit body entirely and let the title carry the question.

The dismiss button is the explicit close affordance. Backdrop click and Escape are conveniences and may be suppressed for high-stakes confirmations (data-modal-static); the visible close button is not. Removing the close button traps the user inside a flow they cannot exit and is incorrect for any non-blocking dialog.

What gets shared?
.modal
The backdrop layer. Owns the scrim, the light-dismiss target, and the entry/exit fade. Fixed-position; dismissed by removing .is-open.
.modal__surface
The dialog box. Owns shape, max-width (size token), max-height cap, and the slide+scale entry transform.
.modal__header
Title region. Composes .modal__eyebrow, .modal__title, .modal__description, and the trailing .modal__close. Add .has-icon when a leading .modal__icon is present.
.modal__body
Primary content slot. Independently scrollable when content overflows the surface height cap.
.modal__footer
Action region. Right-aligned by convention; up to three actions plus a left-aligned .modal__footer-aux for tertiary links.
.modal__close
Always last in the header DOM order. Wires to data-modal-close for the demo behaviour on this page.

02 / Modal — Sizes

Five widths.

Five size caps scale the surface's max-width while the height stays content-driven up to the viewport-relative ceiling (--modal-max-height). XS 320 px, SM 440 px, MD 560 px (default), LG 720 px, XL 960 px.

Size is selected by content, not by emphasis. XS for binary confirmations and short alerts. SM for single-field prompts and one-paragraph notices. MD is the default for confirmations, multi-field prompts, and detail reveals. LG for short forms and dense content that benefits from horizontal room. XL for richer surfaces (preview-and-edit, side-by-side compare). Sizes above XL are not part of this system — promote the flow to a dedicated route instead.

At narrow viewports, every centred modal collapses to a bottom-sheet anchor. Below ~600 px the surface stretches to the viewport width and pins to the bottom edge so the action region stays reachable one-handed. The size token still controls inner padding; only the anchor changes.

03 / Modal — Alert dialog

Confirm before committing.

The alert dialog is a confirmation modal that carries a tonal accent on a leading icon (.modal__icon) and the role="alertdialog" attribute. Reserved for irreversible actions (delete, archive, send, sign-off) where the user must read the consequence before committing. The surface itself stays neutral — only the icon carries the tone.

Tone is selected by intent, not by aesthetics. .modal--tone-critical for destructive or irreversible actions. .modal--tone-warning for caution that does not delete data (sign-off, dispatch, lock). .modal--tone-info for advisory confirmations. .modal--tone-safe for confirmations of completion. A tone selected for visual impact rather than meaning erodes the system's status semantics.

Critical alert dialogs are static. The backdrop and Escape are suppressed (data-modal-static) so the user cannot dismiss a destructive prompt by accident. The visible Cancel action is the only path back. Non-critical alert dialogs follow the regular dismiss conventions.

Action verbs name the destructive outcome. The primary button reads Delete record, not OK or Confirm. The user's last glance before committing should restate exactly what will happen. The Cancel action stays neutral and is always the safer affordance.

04 / Modal — Confirm

A short question.

The confirm dialog is the everyday yes/no surface. Title carries the question, optional description explains the consequence, footer carries the two actions. .modal--compact condenses the spacing for the shortest prompts; the standard layout is used when a description is needed.

The question is in the title, not in the description. The user must be able to answer from the title alone. The description provides context that confirms what will be affected, not whether to proceed.

The primary action sits last (right) by convention. Reading order ends on the affirmative path so the user lands on the action that progresses the flow. The Cancel action takes the secondary slot. Inverting this order leads users to tab into Cancel and dismiss when they meant to proceed.

05 / Modal — Form dialog

A short task.

The form dialog hosts a small, self-contained data entry task — a quick capture, an inline edit, a single-step record. The body slot accepts any form fields from the Form component; the footer carries the submit/cancel pair. Surfaces longer than ~6 fields belong on a dedicated route, not in a modal.

A modal is the wrong place for a long form. Once the body needs to scroll past two viewport heights the user loses the spatial cue of the surrounding page, the focus trap becomes punitive, and field-level errors break the user's mental model of what's visible. Promote the flow to a dedicated route once the field count or required reading exceeds a single glance.

Submit affordance reflects state. The primary action is disabled until the form is valid, enters a loading state during submission (.btn--loading), and the close button is suppressed while the request is in flight so the user cannot dismiss the dialog mid-write.

06 / Modal — Scrollable

Sticky frame.

.modal--scrollable pins the header and footer while the body scrolls inside the surface height cap. Reserved for content the user must navigate (terms, change-log, long lists) where losing the dismiss control on scroll is incorrect. The seam between the sticky slots and the scrolling content is drawn by a hairline divider so the slot boundaries stay legible.

Scrolling content does not replace pagination. Lists with more than ~50 rows belong on a dedicated route with a real list component. The scrollable modal is for content that reads top-to-bottom in a single sitting (terms, audit trail, release notes) — not for lists that need filtering or sorting.

Records-retention policy

Terms of access · v3.2

Records older than seven years are read-only by default. Edit access can be re-enabled by an administrator on a per-record basis, and every edit on a re-enabled record is permanently logged in the audit trail.

Records flagged as part of an active compliance review are locked for the duration of the review window. Lock status is visible at the top of every chart and on the records index.

Sharing a chart externally generates a read-only access link valid for 24 hours. The link is single-use; a new link must be generated for every additional reviewer. The recipient cannot extend or forward the link.

External reviewers see the bloodwork panel, the patient's weight history, and any flagged withdrawal periods. No owner-identifying fields are shared. Sharing events are logged on the chart's activity timeline.

07 / Modal — Drawer & sheet

Edge anchors.

Edge-anchored variants keep the surface attached to one edge of the viewport instead of centring it. Drawer (.modal--drawer-right, .modal--drawer-left): full-height side sheet, animates in horizontally. Bottom-sheet (.modal--sheet): full-width bottom sheet, animates in vertically; rounds only the top corners. Top drawer (.modal--drawer-top): full-width top sheet for system-level interrupts.

Drawers carry context that supplements the underlying view. Filters, settings, detail panels, and helper content where the user needs to see the host page while interacting with the panel. The right drawer is the convention for the primary content reveal; the left drawer is reserved for navigation or secondary panels.

Bottom-sheets are the mobile-first anchor. On touch viewports the bottom-sheet is reachable one-handed and matches the platform convention. On desktop, bottom-sheets are reserved for non-task affordances (cookie notice, recent items) where the underlying page must remain visible.

Drag-to-dismiss requires a handle. A bottom-sheet that supports drag-to-dismiss exposes .modal__handle at the top of the surface so the affordance is visible. Sheets without the handle dismiss only via the close button or backdrop.

08 / Modal — Fullscreen

Cover the viewport.

.modal--fullscreen covers the entire viewport and removes the surface's radius and border. Reserved for immersive flows: image and chart viewers, multi-step wizards, on-call dashboards that must dominate the screen, and on-mobile substitution for any modal that would otherwise overflow the viewport.

Fullscreen modals are still dialogs. They keep the focus trap, the dismiss button, and the header. A fullscreen surface that hides the close affordance becomes a full route — at which point the flow belongs in the router, not in a dialog.

Fullscreen is the small-viewport fallback. Below ~480 px any centred modal that would otherwise need horizontal scroll (preview, side-by-side compare, image viewer) promotes to fullscreen automatically. The desktop surface uses .modal--lg or larger; the same dialog on mobile resolves through .modal--fullscreen.

09 / Modal — Behaviour & a11y

A focus boundary.

A modal is defined as much by its behaviour as by its surface. Focus trap, scroll-lock, return-focus, and the ARIA wiring are not optional — a surface that lacks any of them is not a modal, even if it renders one.

Focus enters the dialog on open and is trapped while it is mounted. The first focusable element in the dialog receives focus (typically the close button for read-only dialogs, the first input for forms, the Cancel action for destructive alert dialogs). Tab and Shift+Tab cycle within the dialog only. Focus returns to the trigger on close.

Body scroll is locked while a modal is open. The page script applies .has-modal-open on <body>; the rule in modal.css sets overflow: hidden. Scrolling inside the modal's body slot is unaffected. Locking the background prevents the user from losing the dialog offscreen and keeps the visual focus on the surface.

ARIA wiring is mandatory. Every dialog uses role="dialog" (or role="alertdialog" for confirmations that announce on open), carries aria-modal="true", and references the title via aria-labelledby. The optional description, when present, is referenced by aria-describedby.
