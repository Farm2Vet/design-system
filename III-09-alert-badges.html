III-09 / Components Alert & Badge.

Two feedback components. Alert is a horizontal strip composed of icon, title, body, optional actions, and optional dismiss. Badge is an inline pill composed of label, optional dot or icon, optional count, and optional close. Both resolve through component-scoped token layers (--alert-*, --badge-*) that reference the system's shared status colour families.

Architectural note · scope boundaries

Alert is the non-blocking, in-flow message surface. A blocking message that interrupts the task flow is a Modal. A transient message that auto-dismisses is a Toast. A status message pinned inside a card's content slots is .card.card--left-bar, documented under III-05-cards.html. Alert covers page-top banners, section-level system messages, validation summaries, and inline guidance attached to form fields.

Architectural note · badge vs. card__badge

The standalone .badge is the canonical pill. The Card component ships its own .card__badge sub-component scoped to the .card__badge-row slot — same visual language, pre-positioned for in-card use. Outside of a card surface (tables, list rows, page headers, alerts, status rails) the standalone .badge is the correct primitive.

01 / Alert — Anatomy

Anatomy.

An Alert is a horizontal strip composed of three regions: a leading icon, a content stack containing title and body, and a trailing region containing actions and a dismiss button. Variants differ only in which slots they expose; the underlying strip is constant.

Title is the only required slot. An Alert without a title resolves as a Compact one-liner (see Layouts). Alerts without an icon are valid for purely textual notices but lose the tonal glance-cue, so use only when the surrounding context already conveys the tone.

02 / Alert — Tone matrix

Tones.

Five canonical tones — Info, Safe, Warning, Critical, Neutral — each remap the alert's surface, border, foreground, and icon token slots. Tones map 1:1 with the system's status colour families and with the Card component's .is-* modifiers.

Tone is selected by intent, not aesthetics. Info for advisories and system messages. Safe for confirmation of a successful state. Warning for non-blocking caution. Critical for failures and blocking errors. Neutral for tips, hints, and notices with no status meaning. A tone selected for visual impact rather than meaning erodes the system's status semantics.

One Alert per intent. Multiple Critical banners stacked together flatten the meaning of Critical. When several issues share a single intent, condense them into one Alert with an .alert__list of sub-issues.

03 / Alert — Style variants

Style.

Four style treatments along an emphasis axis. Soft (default): tinted surface with a thin coloured border. Solid: white-on-tone fill, no border. Outlined: transparent surface with a coloured border. Banner: full-bleed, zero radius, with a leading accent bar (--alert-bar-width) in the tone colour.

Style is selected by surface, not by tone. Soft is the default inside content surfaces. Solid is reserved for page-top banners covering platform-level events (incidents, outages, billing blocks). Outlined is for inline notices in dense forms or dashboards where a tinted surface would compound visual noise. Banner is exclusive to the full-bleed page-top region.

One style per region. Mixing Soft and Outlined Alerts within the same section creates competing emphasis cues. Resolve to a single style per region.

04 / Alert — Layouts

Layouts.

Four layout modes resolve from the same DOM by exposing or hiding slots. Default: title and body. Compact (.alert--compact): title-only, single-line, reduced padding. With actions: trailing .alert__actions group with up to two affordances. Stacked (.alert--stacked): icon, content, and actions arranged vertically. Dismissibility is orthogonal to layout — the close button is a slot, not a variant.

Cap trailing actions at two. The .alert__actions region is sized for one primary (.alert__action--strong) and one secondary affordance. A third action belongs in the surrounding flow, not inside the Alert.

Use .alert__list for validation summaries. When a single Alert needs to surface multiple sub-issues (form validation, batch error report), nest a <ul> with the .alert__list class inside .alert__content. Avoid stacking multiple Critical Alerts to convey the same intent.

Stacked is for narrow columns only. Reach for .alert--stacked when the host column is below ~420 px (sidebars, drawers, modal panels). At wider widths the vertical stack wastes horizontal real estate.

05 / Alert — Sizes

Sizes.

Three sizes scale padding, icon glyph, and title font-size in lockstep. SM: --text-sm font, 16 px icon, compact padding. MD (default): --text-base font, 20 px icon. LG: --text-h6 title, 24 px icon, generous padding. Body line-height is shared (--alert-body-line) across all sizes.

Size is selected by hierarchy, not by tone. SM is reserved for inline guidance attached to a form field or embedded inside another component. MD is the default for section-level Alerts. LG is reserved for the dominant Alert above a container — never more than one LG Alert per view.

06 / Badge — Anatomy

Anatomy.

A Status Badge is an inline pill composed of up to four slots: a leading dot (.badge__dot) or icon (.badge__icon), a label (.badge__label), and an optional trailing close button (.badge__close) for removable chips. The .badge--count modifier turns the same pill into a numeric count badge. Variants compose by adding or removing slots; the underlying shape is constant.

Label content is a noun or short adjective phrase. Status (Active, Pending), role (Owner, Vet), or attribute (New, Premium). Sentences belong in body text, not in a badge.

Dot and icon are mutually exclusive. A badge with both leading slots filled creates a redundant visual cue. Pair the dot with verb-state labels (live status); pair the icon with category or attribute labels.

.badge__dot

.badge__icon

.badge__label

07 / Badge — Tone × Style

Tones.

Seven tones — Neutral, Primary, Info, Safe, Warning, Critical, Accent — compose with four styles. Soft (default): tinted background, transparent border. Solid: opaque tone fill, white foreground. Outlined: transparent background, coloured border. Ghost: transparent background, no border, tone-coloured text only.

Status tones are reserved for status meaning. .is-info always denotes information, .is-safe success, .is-warning caution, .is-critical error. Non-status concepts (Premium, Beta, Featured) take the Neutral, Primary, or Accent tones — never a status colour borrowed for visual impact.

Style is selected by surrounding density. Soft is the default for inline metadata. Solid is reserved for badges that must read at a glance (count badges, severity stamps). Outlined replaces Soft in dense layouts where additional tinted surfaces would compound visual noise. Ghost is for the lowest emphasis read — typically a labelled status dot in a list row.

08 / Badge — Sizes

Sizes.

Four sizes scale font-size, padding, and pill height in lockstep. XS: 10 px font, 16 px height. SM: --text-micro, 20 px height. MD (default): --text-sm, 24 px height. LG: --text-base, 32 px height. Letter spacing and font weight are shared across all sizes.

Size is selected by host context. XS is reserved for badges nested inside other components (count badges, in-row indicators); never use XS as a standalone label. SM is for inline metadata in dense lists or compact rows. MD is the default for badges sitting next to body copy. LG is for badges paired with a heading at H4 or larger.

09 / Badge — Variants & recipes

Variants.

Eight conventional badge shapes composed from the slot vocabulary above: with leading dot, with pulsing dot, with leading icon, removable chip, square monogram, count badge, count-anchored host, and standalone dot indicator.

With leading dot live status

The dot is paired with verb-state labels. Active, Idle, Offline, Awaiting review. Pairing the dot with a non-status noun (Premium, Beta) is incorrect — the dot is a live-state cue, not a generic accent.

Live Idle · 4 m Awaiting review Offline Archived

With leading icon

Icons reinforce; they do not replace the label. Every badge with an icon also carries a textual label. Icons add glanceability for dense rows and improve recognition in multi-language contexts.

Verified Awaiting · 2 h Action needed Premium

Removable / dismissible chip

Removable badges represent user-applied state. Filter pills, multi-select selections, applied tags. The .badge__close button is always visible — never revealed on hover only, since touch surfaces have no hover state.

Square · monogram

Square radius is reserved for codes and identifiers. .badge--square swaps the pill radius for --badge-radius-square. Use for SKU labels, version tags, plate numbers, draft / beta stamps — fixed-format strings where the pill shape would misread as status.

SKU-12-088 v0.5.2 BETA Draft

Count badges numeric

Count badges represent quantitative state. Unread items, queued tasks, overdue records. .badge--count applies tabular numerals, semibold weight, and the count-specific token slots (--badge-count-bg, --badge-count-fg). Wrap the host element in .badge-host to anchor the count in the top-right corner with a 2-px ring against the host surface.

Cap displayed values at three digits. Counts above 999 are rendered as 99+ or 999+. The exact value belongs in a tooltip or the destination view, not in the pill itself. Use .badge--count.is-empty for a presence-only pip when the exact count is irrelevant.

Standalone dot

.badge--dot-only is the minimum status affordance. A bare coloured dot, no label, no pill. Reserved for list rows or cells where a full pill would dominate the row's vertical rhythm. Pair with adjacent text (the row's title) so the dot's meaning resolves from context.

.is-pulse denotes an actively live state. Incidents in progress, on-call status, recording, real-time sync. Animation must not be applied to non-live status (a pulsing "Archived" pip misreads as motion-for-motion's-sake) and is automatically disabled under prefers-reduced-motion.
