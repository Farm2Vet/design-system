Section 07 / Components Toast.

A transient, non-blocking notification surface that floats above the main content. Composed of icon, content stack (title and body), optional actions, optional dismiss, and optional auto-dismiss progress. Resolves through a component-scoped token layer (--toast-*) that references the system's shared status colour families and echoes the Alert family one-for-one.

Architectural note · scope boundaries

Toast is the transient, auto-dismissing feedback surface. A non-blocking message that stays in the document flow is an Alert. A blocking message that interrupts the task flow is a Modal. Toast is reserved for ephemeral confirmations (Saved, Sent), recoverable failures (Sync failed · Retry), and out-of-band system events the user did not directly trigger.

Architectural note · toast vs. snackbar

Snackbar is a sub-variant of Toast, not a separate component. The bottom-centre copy-confirmation pill used across the token-copy pages (primitives, semantics) is the .toast.toast--snackbar variant. It bypasses the slot vocabulary so legacy copy-to-clipboard flows can inject raw text. For anything the user might need to act on, use a notification toast with the full slot DOM.

Architectural note · region and stack

Toasts never live in document flow. They are appended to a .toast-region, a fixed-position container anchored to one of six viewport edges. Inside the region, toasts of the same tone collect in a .toast-stack that collapses to a peek-behind pile so a flood of one tone (a burst of validation errors, a chain of sync failures) cannot grow off-screen.

01 / Toast — Anatomy

A floating strip.

A Toast is a floating horizontal strip composed of three regions: a leading icon (or spinner for promises), a content stack containing title and body, and a trailing region containing actions and a dismiss button. An optional progress bar pinned to the bottom edge visualises the auto-dismiss timer.

Title is the only required slot. A toast without a title resolves as a Compact one-liner (see Layouts). Toasts without an icon are valid for purely textual confirmations but lose the tonal glance-cue, so reserve icon-less toasts for the Snackbar variant where the absence is part of the language.

Default lifetime is five seconds. Toast content must be parseable in that window. If the message exceeds a one-line title plus a one-line body, it is too long for a Toast — promote it to an Alert. Hover and focus pause the auto-dismiss timer; the progress bar reflects the remaining lifetime.

.toast__actions
Optional trailing region for ≤ 1 action (Undo, Retry, View). Two actions belong in an Alert; the toast lifetime is too short for a deliberation.
.toast__close
Always last in DOM order, before the progress bar. Cancels the auto-dismiss timer immediately.
.toast__progress
Optional auto-dismiss progress. Pauses on hover or focus-within. Hidden under prefers-reduced-motion.
02 / Toast — Tone matrix

Five tones.

Five canonical tones — Info, Safe, Warning, Critical, Neutral — each remap the toast's surface, border, foreground, and icon token slots. Tones map 1:1 with the system's status colour families and with the Alert component's .is-* modifiers.

Tone is selected by intent, not aesthetics. Info for advisory confirmations and out-of-band updates. Safe for confirmation of a successful action. Warning for non-blocking caution that does not require an immediate response. Critical for failures the user can still recover from with one action. Neutral for tips, hints, and ambient feedback with no status meaning.

Critical toasts must always carry a recovery action. A Critical toast that disappears with no way to retry, undo, or open the failing record is a dead-end. If the failure has no actionable recovery, the message belongs in an Alert pinned to the affected surface, not in a Toast.

03 / Toast — Style variants

Style by emphasis.

Three style treatments along an emphasis axis. Soft (default): tinted surface with a thin coloured border, tone carried by the surface itself. Solid: white-on-tone fill, no border, reserved for the most urgent recoveries. Inverted: dark glass surface where tone is carried by the leading icon glyph only — the most "OS-level" notification look.

Style is selected once per surface. A product should commit to one style as the house default and use the others only for deliberate exceptions. Mixing Soft, Solid, and Inverted toasts in the same region creates competing emphasis cues and erodes the system's tone semantics.

Inverted is preferred when toasts overlap dense or colourful content. A tinted Soft toast can lose contrast against a chart, a photo, or a coloured banner. The dark glass of .toast--inverted reads cleanly over any background and is the safest default for surfaces that mix text, imagery, and data viz.

04 / Toast — Layouts

Seven shapes.

Seven layout modes resolve from the same DOM by exposing or hiding slots. Default: title and body. Compact (.toast--compact): title-only, single-line. With action: trailing CTA. With dismiss: explicit close × that cancels the auto-dismiss timer. With progress: auto-dismiss progress bar pinned to the bottom edge. Promise / loading: spinner replaces the leading icon. Snackbar: documented separately in section 08.

Cap trailing actions at one. The toast lifetime is too short for a deliberation between two affordances. If a decision is required, the message belongs in an Alert or Modal, not a Toast.

Promise toasts replace themselves rather than stacking. A loading toast announces an in-flight operation; on completion it is upgraded in place to a Safe or Critical toast (same DOM node, swapped class and content). Spawning a new toast on completion would double the noise.

Progress is a visual cue, not a guarantee. The progress bar reflects the auto-dismiss timer, not the state of any underlying operation. Use the spinner of the Promise layout when the toast represents in-flight work.

05 / Toast — Sizes

Three scales.

Three sizes scale padding, icon glyph, title font-size, and minimum width in lockstep. SM: --text-sm font, 16 px icon, compact padding. MD (default): --text-base font, 20 px icon. LG: --text-h6 title, 24 px icon, generous padding. Body line-height is shared (--toast-body-line) across all sizes.

Size is selected by host viewport, not by tone. MD is the default for desktop and tablet. SM is reserved for narrow regions (mobile, side-rail panels) where the default width crowds the content beneath it. LG is for the dominant toast in a marketing or onboarding context — never more than one LG toast on screen at a time.

06 / Toast — Region position

Six anchors.

Toasts live inside a .toast-region, a fixed-position container anchored to one of six viewport edges: top-left, top-centre, top-right, bottom-left, bottom-centre, bottom-right. The anchor is set on the region via data-position; toasts inside the region inherit the entry direction and stack flow from it.

Pick one anchor per product and keep it. Toasts that move between anchors during a session train users to expect inconsistency, and the absence of a fixed gaze target erodes the system's ability to surface urgency.

Right-anchored regions are the safest default for desktop. They sit furthest from primary navigation, do not occlude the document outline on the left, and align with the conventional placement of OS-level notifications. Reach for bottom-centre only for one-line snackbars that confirm a click the user just made.

Top-anchored regions reverse the visual order. In a top region the newest toast renders at the top of the pile so it never has to push older toasts off the viewport edge. In a bottom region the newest toast renders at the bottom, closest to the viewport edge, with older toasts peeking upward.

07 / Toast — Stacking

Same-tone grouping.

Toasts of the same tone collect inside a .toast-stack. The stack is a simple vertical column — toasts share a common gap and follow the region's enter and exit motion. Grouping by tone keeps the glance-cue of each channel intact: a Critical pile and a Safe pile read as separate streams of feedback rather than one mixed column.

One stack per tone, per region. Mixing tones inside a single stack defeats the glance-cue of the front toast. A burst of mixed tones surfaces as multiple stacks one above the other inside the same region — never as a heterogeneous pile.

Coalesce repeats before queuing them. Five identical "Sync failed" toasts should resolve as one "Sync failed · 5 attempts" toast with a single recovery action. Stacks are for distinct events, not for retries of the same event.

Cap concurrent toasts per region. A region with more than four or five visible toasts has stopped communicating priority — every message flattens into noise. When the live count would exceed the budget, dismiss the oldest toast (FIFO) before mounting the next one.

08 / Toast — Snackbar variant

A bare pill.

The Snackbar (.toast.toast--snackbar) is a single-line, bottom-centre pill reserved for ephemeral confirmations that have no actionable consequence — a clipboard copy, a draft save, an undo expiry. It bypasses the slot vocabulary so legacy copy-to-clipboard flows (token-copy pages, share-link buttons) can inject raw text without rewriting the host markup.

Snackbar is for confirmations the user already knows about. The user just clicked Copy; the snackbar acknowledges it. The user just pressed Cmd-S; the snackbar acknowledges it. Use a notification toast (with icon, body, and optional action) for messages the user did not directly trigger — system events, background syncs, out-of-band updates.

One snackbar at a time. A second snackbar replaces the first in place — they never stack. The pill is a single-instance element bound to one DOM node and animated via the .show class.

09 / Toast — Component tokens

Token map.

The component token layer for Toast. Each component token references a semantic token (never a primitive directly), so changes to a status colour family or surface token propagate through every toast consistently. Tone tokens echo the Alert family one-for-one; the two components share a single status colour vocabulary.
