III-08 / Components Toast.

A transient, non-blocking notification surface that floats above the main content. Composed of icon, content stack (title and body), optional actions, optional dismiss, and optional auto-dismiss progress. Resolves through a component-scoped token layer (--toast-*) that references the system's shared status colour families and echoes the Alert family one-for-one.

Architectural note · scope boundaries

Toast is the transient, auto-dismissing feedback surface. A non-blocking message that stays in the document flow is an Alert. A blocking message that interrupts the task flow is a Modal. Toast is reserved for ephemeral confirmations (Saved, Sent), recoverable failures (Sync failed · Retry), and out-of-band system events the user did not directly trigger.

Architectural note · region

Toasts never live in document flow. They are appended to a .toast-region, a fixed-position container anchored to one of two viewport edges (top-right default, bottom-right). New toasts are appended to the region's flex column in DOM order; bottom-anchored regions reverse the visual flow so the newest toast still sits closest to the viewport edge.

01 / Toast — Anatomy

Anatomy.

A Toast is a floating horizontal strip composed of three regions: a leading icon (or spinner for promises), a content stack containing title and body, and a trailing region containing actions and a dismiss button. An optional progress bar pinned to the bottom edge visualises the auto-dismiss timer.

Title is the only required slot. A toast without a title resolves as a Compact one-liner (see Layouts). Toasts without an icon are valid for purely textual confirmations but lose the tonal glance-cue — reach for the Compact layout instead, which keeps the icon while trimming everything else.

Default lifetime is five seconds. Toast content must be parseable in that window. If the message exceeds a one-line title plus a one-line body, it is too long for a Toast — promote it to an Alert. Hover and focus pause the auto-dismiss timer; the progress bar reflects the remaining lifetime.

Always last in DOM order, before the progress bar. Cancels the auto-dismiss timer immediately.

Optional auto-dismiss progress. Pauses on hover or focus-within. Hidden under prefers-reduced-motion.
02 / Toast — Tone matrix

Tones.

Five canonical tones — Info, Safe, Warning, Critical, Neutral — each remap the toast's surface, border, foreground, and icon token slots. Tones map 1:1 with the system's status colour families and with the Alert component's .is-* modifiers.

Tone is selected by intent, not aesthetics. Info for advisory confirmations and out-of-band updates. Safe for confirmation of a successful action. Warning for non-blocking caution that does not require an immediate response. Critical for failures the user can still recover from with one action. Neutral for tips, hints, and ambient feedback with no status meaning.

Critical toasts must always carry a recovery action. A Critical toast that disappears with no way to retry, undo, or open the failing record is a dead-end. If the failure has no actionable recovery, the message belongs in an Alert pinned to the affected surface, not in a Toast.

03 / Toast — Layouts

Layouts.

Six layout modes resolve from the same DOM by exposing or hiding slots. Default: title and body. Compact (.toast--compact): title-only, single-line. With action: trailing CTA. With dismiss: explicit close × that cancels the auto-dismiss timer. With progress: auto-dismiss progress bar pinned to the bottom edge. Promise / loading: spinner replaces the leading icon.

Cap trailing actions at one. The toast lifetime is too short for a deliberation between two affordances. If a decision is required, the message belongs in an Alert or Modal, not a Toast.

Promise toasts replace themselves rather than stacking. A loading toast announces an in-flight operation; on completion it is upgraded in place to a Safe or Critical toast (same DOM node, swapped class and content). Spawning a new toast on completion would double the noise.

Progress is a visual cue, not a guarantee. The progress bar reflects the auto-dismiss timer, not the state of any underlying operation. Use the spinner of the Promise layout when the toast represents in-flight work.

04 / Toast — Sizes

Sizes.

Three sizes scale padding, icon glyph, title font-size, and minimum width in lockstep. SM: --text-sm (13.33 px) font, 16 px icon, compact padding. MD (default): --text-base (16 px / 1 rem) font, 20 px icon. LG: --text-base (16 px) body with a --text-h6 (19.2 px) title, 24 px icon, generous padding. Body line-height is shared (--toast-body-line) across all sizes.

Size is selected by host viewport, not by tone. MD is the default for desktop and tablet. SM is reserved for narrow regions (mobile, side-rail panels) where the default width crowds the content beneath it. LG is for the dominant toast in a marketing or onboarding context — never more than one LG toast on screen at a time.

05 / Toast — Region position

Position.

Toasts live inside a .toast-region, a fixed-position container anchored to one of two viewport edges: top-right (default) and bottom-right. The anchor is set on the region via data-position; toasts inside the region inherit the entry direction and stack flow from it.

Pick one anchor per product and keep it. Toasts that move between anchors during a session train users to expect inconsistency, and the absence of a fixed gaze target erodes the system's ability to surface urgency.

Right-anchored regions are the safest default for desktop. They sit furthest from primary navigation, do not occlude the document outline on the left, and align with the conventional placement of OS-level notifications. Pair the default top-right with a bottom-right alt only when the product surface has a persistent header that would occlude the top row.

Top-anchored regions reverse the visual order. In a top region the newest toast renders at the top of the pile so it never has to push older toasts off the viewport edge. In a bottom region the newest toast renders at the bottom, closest to the viewport edge, with older toasts peeking upward.

06 / Toast — Component tokens

Tokens.

The component token layer for Toast. Each component token references a semantic token (never a primitive directly), so changes to a status colour family or surface token propagate through every toast consistently. Tone tokens echo the Alert family one-for-one; the two components share a single status colour vocabulary.
