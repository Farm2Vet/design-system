Design Tokens

1.1  Token Hierarchy & Usage Guidelines

The Farm2Vet token system uses a strict three-layer hierarchy. Each layer has a single responsibility. Never skip layers — a component should never reference a primitive directly.

✓ Correct usage
  • →Component code references --background-primary
  • →Semantic tokens use var(--p400) or color-mix()
  • →New components use semantic layer tokens directly
  • →Legacy components use component aliases (migration path)
  • →Themed surfaces built exclusively from semantic tokens
✕ Never do this
  • →Using --p500 or #0F3A24 in a component
  • →Hardcoded rgba() or hex values in the semantic layer
  • →Creating semantic tokens that don't reference primitives
  • →Mixing layers — e.g. color-mix(var(--p500), #fff)
  • →Bypassing semantic tokens by redefining colors inline
ℹ
Layer rules: Primitive → raw named values only. Semantic → must reference primitives via var() or color-mix(). Component → maps semantic slots to named component properties. Component aliases exist for backward compatibility — new components should use semantic tokens directly.

Usage Guidelines

3.1  Glassmorphism — Usage Constraints
✓  Allowed — marketing layer only
  • →Hero section overlay (photographic background behind)
  • →Floating navigation pill when scrolled over hero image
  • →Modal backdrop on dark-themed promotional sections
  • →Landing page feature cards (over colored/photo bg only)
✕  Avoid — replace with solid surface
  • →All product-layer cards (use --background-default)
  • →Diagnosis result cards — solid surface mandatory
  • →CTA / action buttons — solid primary only
  • →Form inputs — solid border + background
  • →Data tables, health records, case lists
  • →Alert and status banners

--glass-blurValues: blur(12px), blur(24px). Sets lower value for low-end devices.
