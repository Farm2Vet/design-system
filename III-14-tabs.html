III-14 / Components Tabs.

Tabs select one-of-N peer views of the same content area. The selected tab maps to a panel rendered directly below the list; if there is no panel, the contract is no longer Tabs. Five visual variants — Line, Enclosed, Pill, Segmented, Vertical — share a single token chain and a single keyboard model. The page also documents two sibling navigation primitives that share Tabs' layout zone: .breadcrumb for path display and .pagination for numeric step navigation.

01 / Tabs — Scope

Scope.

Before reaching for Tabs, confirm the contract. Tabs are for switching between peer views inside a single page. They are not for navigating between pages, not for filtering a list, and not for showing where the user is in a hierarchy. Each of those tasks has its own component in this system.

If there is no panel, it is not a tab. The defining contract of Tabs is that selecting a tab mounts a panel of content directly below the list. A control that filters the same content in place is a segmented switch — see .btn-switch--segmented in Buttons. A row of links to other pages is the global header — see Header & Nav Bar.

Component Purpose Reach for it when
.tabs Switch between peer views of the same content area. A single page exposes several views of one record (Overview, Activity, Files), or one tool exposes several modes that share the same canvas (Code, Preview, Issues). Selection mounts a panel.
.breadcrumb Show the user's location in a hierarchy. The user has navigated several levels deep and needs a path back. Breadcrumb is read-only context, not a control surface; the only interactive nodes are the trail links, not the current item.
.pagination Step through a paged result set. A list, table, or feed is sliced into numbered pages. The viewport stays constant; only the data changes. Use compact pagination in dense table footers.
.btn-switch--segmented Pick one-of-N values without mounting a panel. A unit toggle (kg / lb), a sort key (Newest / Oldest), or a one-of-N filter that re-orders the same list in place. The selection is a value, not a view.
.global-nav Navigate between pages of the product or system. The destination is a different URL. The header is the system's only cross-page surface; tabs must never cross page boundaries.
02 / Tabs — Anatomy

Anatomy.

The component is a wrapper, a list, one or more tabs, an indicator, and one panel per tab. Slots are addressable by BEM-style class hooks; ARIA roles are required and enforce the keyboard model.

Panel — content for the selected tab is rendered here.
A · root
The wrapper (.tabs) carries the variant and size modifiers. It owns the local CSS variables that drive the size scale, so child rules never need to know which size is active. Required data-tabs attribute marks the element for the controller to bind.
B · list
.tabs__list with role="tablist". Hosts the tabs and the absolutely-positioned indicator. aria-label describes the group; if the list is preceded by a heading, use aria-labelledby instead.
C · tab
.tabs__tab with role="tab", aria-selected, and aria-controls pointing to its panel. Optional .tabs__icon, .tabs__label, and .tabs__badge sit inside in fixed order so screen readers always announce the same sequence.
D · indicator
.tabs__indicator is the moving primary marker. The controller writes --_x / --_w (horizontal) or --_y / --_h-active (vertical) onto the host so the indicator transitions between tabs without re-rendering the DOM. Hidden in Pill and Enclosed variants.
E · panel
.tabs__panel with role="tabpanel", an id matching the controlling tab's aria-controls, and tabindex="0" so the panel itself is focusable for screen readers. Exactly one panel per tab; orphan tabs are an authoring error.
03 / Tabs — Variants

Variants.

Five visual variants. Every variant honours the same ARIA roles, the same keyboard model, and the same token chain — only the surface treatment and the indicator strategy change. Picking a variant is a layout decision, not a behavioural one.

Match the variant to the surface, not to the importance of the choice. Line is the product default — quiet underline, reads on any background. Enclosed is reserved for tabs that must visually attach to a bordered card. Pill belongs on inverted or glass surfaces where an underline gets lost. Segmented is the treatment of choice for tab strips of three to five short labels in a toolbar. Vertical is for settings panels and doc layouts where labels are long enough to justify the column.

04 / Tabs — States

States.

Rest, hover, focus-visible, selected, disabled. Identical state names, semantics, and tokens across every variant — so a tab in Pill mode and a tab in Line mode read as the same component to the user.

Disable the tab, not the panel. A tab whose panel cannot yet be opened (loading, permission-gated, empty) takes the disabled attribute on the tab itself. Do not render the tab and silently no-op the click; the disabled visual is the contract that explains why the panel is not available. If the reason is permission-driven, pair the disabled tab with a tooltip from the toast layer.

05 / Tabs — Behaviour

Behaviour.

Two activation modes, one keyboard model. Pick the activation mode that matches the cost of mounting a panel; the keyboard contract does not change.

Default to manual activation. Arrow keys move focus between tabs without selecting; the user activates the focused tab with Enter or Space. Switch to automatic activation only when each panel is trivially cheap to render (no network, no heavy computation, no scroll-position reset). Automatic activation on a panel that takes 200ms to mount makes the keyboard model feel laggy and surprising.

Never URL-bind tab selection by default. Tab state is page-level state, not navigation. If the selected tab MUST survive a refresh (deep-linkable record views), encode it as a hash (#activity) — never a path segment. A hash keeps the tab change inside the page-level concept of state and avoids colliding with the global header's contract.

06 / Sibling — Breadcrumb

Breadcrumb.

A breadcrumb shows the user's current location in a hierarchy as an ordered trail of links plus a final non-linking current item. Documented on this page because consumers reach for breadcrumb and tabs in the same layout zone — but the contracts do not overlap.

Breadcrumb is read-only context. The trail items are links; the current item is not. Do not render the current item as a link to itself, and do not give it a hover state — both patterns mislead the user into thinking the current item is a navigable target. If the trail has more than five segments, collapse the middle with a single .breadcrumb__item--ellipsis that opens a menu of the elided segments.

07 / Sibling — Pagination

Pagination.

Pagination is a numeric step navigator for paged result sets. It shares Tabs' toolbar zone and its segmented-list visual contract, but it does not mount panels — it shifts the data underneath an unchanged viewport.

Always show the current page, the first, and the last. The middle elides with a single ellipsis row. Hide the previous button on page 1 and the next button on the last page rather than rendering them disabled — disabled controls in a recurring footer chrome read as a defect, not a contract. For dense table footers, reach for the compact variant.

08 / Tabs — Component tokens

Tokens.

The full --tabs-*, --breadcrumb-*, and --pagination-* token maps, grouped by surface layer.

Override at the component-token layer, never at the call site. Tab tokens reference Form, Button, and Dropdown semantics so the four input families read as one. Breadcrumb and Pagination tokens are intentionally narrow — both are quiet surfaces that should never feel louder than the page they live on. Tune --tabs-indicator-easing once and every sliding indicator across the product re-times together.
