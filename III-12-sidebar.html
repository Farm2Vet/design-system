III-12 / Components Sidebar.

The persistent in-app navigation surface. A vertical rail anchored to the viewport's leading edge that exposes the destinations of a single product — not the structure of the design system itself, which is owned by the Header & Navigation Bar (Section 10). Resolves through a component-scoped --sidebar-* token namespace into the system's neutral surface, primary-tinted active state, and mono-eyebrow group labels.

Architectural note · scope boundaries

The sidebar is the product-level navigation surface. It exposes the destinations of an application consuming this design system — the Wiki, the Chatbot, the Hero mockups, or any future product. The Header & Navigation Bar (Section 10) exposes the documentation system itself and is rendered above every page in this repository; it is not a sidebar and does not collapse to a rail. The Side Panel (Section 07) is a transient drawer raised from a trigger to hold a focused sub-task; it is not a navigation surface and does not persist between routes.

Architectural note · single active item

One item, and only one, carries .is-current at any time. The active state reflects the application's current route. Highlighting two items at once — typically because a parent group and a child sub-item are both marked active — produces an ambiguous reading of location and erodes the rail's usefulness as a wayfinding device. When a sub-item is active, mark only the sub-item; the parent group's .is-open state already communicates that the active route belongs to that group.

01 / Anatomy

Anatomy.

The rail (.sidebar) is a flex column with three vertical regions: a header slot (.sidebar__header) carrying the brand and the collapse trigger, a scrollable nav region (.sidebar__nav) holding mono-labelled groups of items, and a pinned footer slot (.sidebar__footer) for account or utility links. Items render as a three-column row — leading icon, label, trailing slot — and reuse the system's primary-tinted convention for the active state.

Header and footer slots are optional; the nav region is required. A sidebar without a header may be appropriate when the product's brand is already rendered in a top bar above the rail. A footer may be omitted when no utility row is needed. The nav region is the rail's reason for existing — a sidebar with no items is a layout primitive, not this component.

Items are real anchors or buttons. A destination is an <a>; a disclosure is a <button>. Both share the .sidebar__link visual contract so the rail reads as a single uniform list. Wrapping the row in a <div> with a click handler removes the row from the tab order and breaks keyboard navigation.

Page content

The application's route renders here. The rail is rendered once, at the application shell level; the page below changes as the user navigates. Anything that scrolls with the page belongs to the page; anything that persists across routes belongs to the rail.

  • .sidebar The rail container. Owns the column width (--sidebar-width-expanded / --sidebar-width-rail), the leading border, and the collapse transition. Mounts once at the application shell level; rendering it inside a page route detaches it from the focus order on navigation.
  • .sidebar__header Brand slot plus the collapse trigger. Optional — may be omitted when a separate top bar already carries the brand. The collapse trigger is a <button>; its aria-pressed attribute is kept in sync with .sidebar--collapsed.
  • .sidebar__nav The required scrollable region. A <nav> with a meaningful aria-label (typically Primary). Holds one or more groups; long lists scroll inside this region while header and footer remain pinned.
  • .sidebar__group A label plus a list. Groups are visually divided by spacing (--sidebar-group-gap), not by horizontal rules — keeps the rail visually quiet at the destination scale typical of a single product. Use a group only when two or more items share a taxonomy; a single-item group is noise.
  • .sidebar__group-label Mono-eyebrow caption. Names the group's domain (Workspace, Records, Settings). Never carries colour tone — the label is a wayfinding heading, not a status. In collapsed mode the label is replaced by a thin divider.
  • .sidebar__item One row in the rail. Always a <li>; its .sidebar__link child is either an <a> for a destination or a <button> for a disclosure trigger. The visual contract is identical for both.
  • .sidebar__icon Leading 20-px glyph. Required on top-level items for parity in collapsed mode (where the icon becomes the entire row). May be omitted on sub-items, which are anchored by indent rather than icon.
  • .sidebar__trailing Optional trailing slot. Holds a count badge, a status dot, or a disclosure chevron. Only one trailing element per row — stacking a badge and a chevron on the same item produces a noisy edge.
  • .sidebar__footer Pinned bottom slot. Reserved for account, support, and other utility rows that exist outside the primary information architecture. Items inside the footer share the same .sidebar__link contract as the nav region above.
02 / Modes

Modes.

The rail resolves into one of three modes. Expanded is the default and renders the icon plus the label. The rail mode (.sidebar--collapsed) narrows the column to icon-only and exposes the label as a hover tooltip. The overlay mode (.sidebar--overlay) detaches the rail from the page grid and slides it over content, used below the responsive breakpoint when horizontal room is scarce.

Pick a mode by viewport, not by aesthetic preference. Expanded is the default at desktop widths. The rail mode is selected by the user via the collapse trigger and persists as a preference. Overlay is selected by the application below the responsive breakpoint; it is not a desktop affordance.

The label is never removed from the DOM in rail mode. The collapsed rail visually clips the label, but the link's accessible name still resolves to its text content. Replacing labels with aria-label on the icon — a common shortcut — duplicates the label in screen-reader output when the rail expands and is incorrect.

03 / Density

Density.

Density modifies the row's vertical padding and minimum height while preserving the icon size and label type scale. Compact suits power-user surfaces with a long destination list. Default is the canonical density. Comfortable suits surfaces seen from a distance or on touch-first devices.

Density is selected per application, not per group. Mixing densities inside one rail produces a ragged left edge and erodes the rail's read as a single list. The modifier sits on the rail (.sidebar), not on individual groups or items.

04 / States

Rest, hover, current, disabled.

Item rows resolve into four interactive states. The active state — applied to exactly one row at a time — reuses the system's primary-tinted convention so the rail's selection language matches the header bar's and the dropdown menu's. Hover is the neutral background-default-hover; rest is transparent.

Disabled is for routes the user cannot reach from the current context. A route is not disabled because it has no data — it is disabled because the user lacks permission, the feature is gated, or a prerequisite step has not been completed. For an empty state, route to the destination and let the page render its empty view.

Badge tone follows the system's status semantics. Neutral counts use the default .sidebar__badge. --info for advisory counts (drafts, pending). --critical for unresolved alerts that need action. The dot variant replaces a badge when the precise count is not meaningful — only that something new has arrived.

05 / Disclosure

Disclosure.

An item with children renders a trailing chevron and toggles its .sidebar__sublist on click. Sub-items inherit the row contract but drop the leading icon and shift their text to --text-body-sm; an indent rail anchors the visual hierarchy. Disclosure is independent per group — opening one parent does not close another. Nesting is capped at one level.

Use disclosure only when sub-items belong to a single named domain. If a parent's children are heterogeneous, promote them to siblings of the parent under a new group label. Disclosure is a grouping device for the rail; it is not a substitute for a third level of information architecture, which belongs to the page.

Two levels, never three. A sub-item's children belong to the page below the rail, typically as Tabs (Section 14) or a Side Panel (Section 07). Nesting a third level inside the sidebar produces a column wider than the page benefits from and an orientation problem (which level am I on?) the rail cannot resolve.

06 / Behaviour

Open, dismiss, follow.

Pointer behaviour is owned by the application's sidebar runtime; this page documents the contract. The rail's CSS resolves visual state from class hooks (.sidebar--collapsed, .sidebar--overlay, .sidebar__item.is-open, .sidebar__link.is-current); the runtime is responsible for setting and clearing them in step with the application's route and the user's preferences.

Input Outcome
Item link clicked The runtime navigates to the link's href. On the destination route, the runtime resolves the matching item and adds .is-current plus aria-current="page". Any prior current item is cleared. If the matched item is a sub-item, the runtime opens its parent group (sets .is-open and aria-expanded="true").
Disclosure trigger clicked The runtime toggles .is-open on the parent .sidebar__item and updates aria-expanded. The sub-list appears/disappears via CSS transition; multiple groups may be open at once.
Collapse trigger clicked The runtime toggles .sidebar--collapsed on the rail and updates aria-pressed on the trigger. The choice persists in localStorage under a single application-scoped key; pages do not author per-page collapse state.
Hover on item (rail mode) The runtime exposes the item's label as a tooltip pinned to the right edge of the row, referenced by aria-describedby on the link. The tooltip dismisses on mouseleave or blur.
Viewport drops below the responsive breakpoint The runtime swaps the rail into overlay mode (.sidebar--overlay) and hides it behind a trigger in the page's top bar. Opening the rail adds .is-open; the runtime renders a scrim and traps focus inside the rail until dismissed.
Outside click (overlay mode) The runtime closes the rail and returns focus to the trigger that opened it. Clicks on items inside the rail navigate normally; the rail dismisses itself after the navigation commits.
Esc pressed (overlay mode) The runtime closes the rail and returns focus to the trigger. Esc is a no-op in expanded and rail modes — those are persistent surfaces, not dismissable ones.
Page scrolls The rail does not scroll with the page. Internal overflow is owned by the .sidebar__nav region, which scrolls independently while header and footer remain pinned.
07 / Accessibility

Roles, labels, focus.

The rail uses native semantics where possible: a <nav> with a meaningful label, real <ul> / <li> markup for the list, and <a> for destinations alongside <button> for disclosures. ARIA fills the gaps the markup cannot — current page, expansion state, disabled-but-focusable items.

  • Container The outer element is an <aside> carrying a contextual label (aria-label="Primary" for the application's top-level rail). The internal nav region is a <nav> with its own label naming the contained list (Primary destinations, Records).
  • Active item The active row carries aria-current="page" on the anchor. The class hook .is-current drives the visual state; the ARIA attribute is the source of truth for assistive tech. Both must be set together.
  • Disclosure A parent with children is rendered as a <button type="button">, not an anchor. aria-expanded is kept in sync with the parent .sidebar__item.is-open state by the runtime. The sub-list does not need role — it is a real <ul>.
  • Disabled items Carry aria-disabled="true" rather than the disabled attribute, which is not valid on anchors. The CSS suppresses pointer events; the row remains in the tab order so the user can still hear the destination's name and understand why it is unreachable.
  • Collapse trigger A toggle button. aria-pressed reflects the rail's collapsed state; aria-label reads Collapse sidebar when expanded and Expand sidebar when collapsed.
  • Rail-mode tooltips Each tooltip carries role="tooltip" and is referenced by aria-describedby on its parent link. The label remains in the link's accessible name — it is visually clipped, not removed —so the link continues to announce its destination correctly when the rail expands.
  • Keyboard Tab moves through items in DOM order. Enter activates a link; Space activates a button. Esc dismisses the rail in overlay mode and is a no-op in the persistent modes. No custom arrow-key behaviour is added — the rail is a list of links, not a roving menu.
  • Visible focus Focus rings inherit the system's :focus-visible ring from base.css. The rail does not suppress focus styles in any mode.
  • Reduced motion The rail's collapse, the overlay slide, and the chevron rotation respect the system's prefers-reduced-motion rule from base.css. Transitions are shortened to instant; no rail-specific animation is added beyond the shared transition.
08 / Responsive

Three breakpoints.

The rail collapses in two stages. Above 1024 px the rail renders expanded by default; the user may collapse it to the rail mode via the trigger and the choice persists. Between 720 px and 1023 px the application defaults to rail mode and the trigger is hidden — horizontal room is too tight for the expanded width to coexist with primary content. Below 720 px the rail switches to overlay and is hidden behind a trigger in the page's top bar.

  • ≥ 1024 px Expanded rail (264 px) by default. User may collapse to the 64 px rail mode; choice persists per application via localStorage.
  • 720–1023 px Forced rail mode. Collapse trigger hidden because there is no other state to switch to. Tooltips on hover expose item names.
  • ≤ 719 px Overlay mode. Rail is hidden by default; opened from a trigger in the page's top bar. Outside click and Esc dismiss.
09 / Usage

Mount once, route freely.

The rail is rendered once at the application shell level, outside the page route. Pages do not author per-link active classes; the runtime resolves the active route on each navigation and sets .is-current plus aria-current on the matching item. 

<!-- 1. Container — anchor at the application shell, not in a page route -->
<aside class="sidebar" aria-label="Primary">

  <!-- 2. Header slot — optional -->
  <header class="sidebar__header">
    <a href="/" class="sidebar__brand">…</a>
    <button type="button" class="sidebar__collapse"
            aria-label="Collapse sidebar" aria-pressed="false">…</button>
  </header>

  <!-- 3. Nav region — required -->
  <nav class="sidebar__nav" aria-label="Primary destinations">

    <div class="sidebar__group">
      <p class="sidebar__group-label">Workspace</p>
      <ul class="sidebar__list">
        <li class="sidebar__item">
          <a class="sidebar__link" href="/overview">
            <span class="sidebar__icon" aria-hidden="true">…</span>
            <span class="sidebar__label">Overview</span>
          </a>
        </li>
      </ul>
    </div>

  </nav>

  <!-- 4. Footer slot — optional -->
  <div class="sidebar__footer">…</div>

</aside>

Do not author per-link .is-current classes inside page templates. The runtime is the single source of truth for the active route. Hard-coded active classes drift out of sync with the router and produce two conflicting highlights.

Do not nest a second navigation surface inside the rail. Tabs (Section 14) and breadcrumbs belong to the page below the rail. Putting a horizontal tab strip inside .sidebar__nav produces a cross-axis hierarchy the user cannot reason about.

Do not raise modal content inside the rail. A persistent navigation surface is the wrong host for transient or blocking content. Use the Modal Dialog (Section 06) for blocking interruptions, the Side Panel (Section 07) for focused sub-tasks, and the Toast (Section 08) for transient confirmations.

10 / Component tokens

Tokens.

The rail introduces its own --sidebar-* token namespace, scoped to this component. Every value resolves to a semantic token; no primitive (--neutral-*, --primary-*) is referenced directly. Renaming any consumed semantic token propagates through the rail in step with the rest of the system.
