III-12 / Components Sidebar.

The persistent in-app navigation surface for a single product instance. A glass rail anchored to the viewport's leading edge, exposing the top-level destinations available to the signed-in user.

Architectural note · scope boundaries

The sidebar is the product-level navigation surface. It exposes the destinations of an application consuming this design system — the Wiki, the Chatbot, the Hero mockups, or any future product. The Header & Navigation Bar (Section 10) exposes the documentation system itself and is rendered above every page in this repository; it is not a sidebar and does not collapse to a rail. The Side Panel (Section 07) is a transient drawer raised from a trigger to hold a focused sub-task; it is not a navigation surface and does not persist between routes.

Architectural note · single active item

One item, and only one, carries .is-current at any time. The active state reflects the application's current route. Highlighting two items at once — typically because a parent group and a child sub-item are both marked active — produces an ambiguous reading of location and erodes the rail's usefulness as a wayfinding device. When a sub-item is active, mark only the sub-item; the parent group's .is-open state already communicates that the active route belongs to that group.

01 / Anatomy

Anatomy.

Width, item height, and icon size are fixed component-token values. Above the responsive breakpoint the rail is persistent and the user toggles between expanded and compact; below it, the rail detaches and renders as an overlay drawer. Everything else — header height, item row, icon size, padding, avatar — is identical across viewports.

Icon stroke and grid live with the icon set itself; once IV-02 / Icons is authored the 20 px rail icon will reference that page's canonical icon scale. Until then the sidebar's icon size is authored locally as --sidebar-item-icon-size.

The rail always fills the viewport's height. The component declares min-height: 100vh; the application's shell layout supplies any outer padding or margin (a header above, a margin around the page) and the rail stretches to whatever's left. The rail's height does not change when an item is disclosed, when the user collapses to compact, or when the route changes.

Width is fixed per mode; the rail does not fluid-scale between breakpoints. Expanded is 280 px, compact is 64 px, overlay raises to 280 px. The viewport selects the mode; the mode selects the width. There is no intermediate width and no resize handle — a fluid rail would erode the grid the page below it is laid out against.

02 / Modes

Modes.

The rail resolves into one of two persistent modes. Expanded is the default at desktop widths and renders the icon plus the label. Compact narrows the rail to the icon column and exposes the label as a tooltip on hover; the choice persists per user as an application preference.

Compact is a user preference, not a layout breakpoint. The user opts into the compact rail via the collapse trigger. Overlay (described in §05) is selected by the application below the responsive breakpoint and is not a desktop affordance.

The label is never removed from the DOM in compact mode. The compact rail clips the label visually but keeps it in the link's accessible name. Substituting aria-label on the icon duplicates the label in screen-reader output when the rail re-expands.

Pick the tone by what the rail sits over, not by user preference. Light is the default. Dark is selected by the application when the rail mounts over an image, a hero photograph, or a dark-themed shell. A user-controlled dark mode swaps both the rail and the page beneath it through the system's theme rail, not through this modifier.

Count pips persist into compact mode. A badge marked .sidebar__badge--pip stays visible when the rail is collapsed, repositioned as a small numeric dot at the top-right of the leading icon. Use this only for time-sensitive, unread counts (notifications, chat, alerts). Routine trailing counts (e.g. Visits 6) omit the modifier and hide cleanly when the rail collapses.

03 / States

States.

Item rows resolve into four interactive states. The active state — applied to exactly one row at a time — reuses the system's primary-tinted convention so the rail's selection language matches the header bar's and the dropdown menu's. Hover is the neutral background-default-hover; rest is transparent.

Disabled is for routes the user cannot reach from the current context. A route is not disabled because it has no data — it is disabled because the user lacks permission, the feature is gated, or a prerequisite step has not been completed. For an empty state, route to the destination and let the page render its empty view.

Badge tone follows the system's status semantics. Neutral counts use the default .sidebar__badge. --info for advisory counts (drafts, pending). --critical for unresolved alerts that need action. The dot variant replaces a badge when the precise count is not meaningful — only that something new has arrived.

04 / Component tokens

Component tokens.

The rail introduces its own --sidebar-* token namespace, scoped to this component. Every value resolves to a semantic token; no primitive (--neutral-*, --primary-*) is referenced directly. Renaming any consumed semantic token propagates through the rail in step with the rest of the system.
