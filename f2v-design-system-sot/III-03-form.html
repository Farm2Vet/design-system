III-03 / Components Form.

Eight input primitives that share a single shape, scale and state vocabulary — text, textarea, dropdown, date, multi-select, radio, file upload and checkbox. Each resolves through the same --form-* token layer so a tweak to focus colour or radius cascades across every control.

Pair every input with a visible label. Placeholder text disappears the moment the user types and is invisible to most assistive technology — it can carry an example (e.g. Bessie) but never an instruction. Format constraints, ranges, and unit hints belong in helper text below the field. Match the input type to the data — date, email, tel, number — so platforms surface the right keyboard, picker, and validation. Required fields communicate twice: the visible * for sighted readers and aria-required="true" (or the required attribute) for screen readers. Errors live below the field with a specific, actionable message — never a silent red border.

Use a radio group when exactly one option from a small fixed set must be chosen. Pre-select the safest default — leaving the group empty forces a decision before the user knows the cost of each choice. Group every option under a single legend or visible label, and keep the option labels mutually exclusive. Reach for the bordered list variant once labels carry explanation or the set grows past four options; the inline group is for short, single-word choices.

01b / Form — Search field

Search field.

A composed pattern, not a new component. The wrapper holds a leading magnifier slot, the canonical .input, and a trailing clear-button slot that auto-reveals once the input has a value. Inherits the input token rail end-to-end — apply .input--sm, .input--md, etc. on the .search-field__input just as you would on a bare .input.

Empty · rest
:placeholder-shownicon · input · no clear
Active · with value
:not(:placeholder-shown)clear button auto-reveals

Use .search-field when search is the primary action of a region. Toolbars, modal headers, and dense list filters all justify the composed wrapper — the leading magnifier signals affordance from across the page, the clear-button slot lets the user reset without selecting and deleting. For a single free-text filter buried inside a longer form (one of many fields, with its own label), use a bare .input — the wrapper would crowd the form rhythm and duplicate the field's label intent.

III-03 / NN

Padding ratio.

Text fields obey the golden band at the canonical 16px size: vertical padding runs 0.5 – 0.75 of the input font, and horizontal padding runs 1.0 – 1.5 of it — resolving to 8–12px vertical and 16–24px horizontal. Above the canonical size the rail diverges by convention: X-padding caps at 32px for XL and 2XL because inputs don't hug their value the way buttons hug their label, and scaling X further only pushes the placeholder away from the left edge into empty rail. Below the canonical, SM sits as a density / utility tier below the Y floor by design — the same pattern Bootstrap, Polaris, Mantine, Ant, and shadcn all follow for their small inputs.

Golden band — form rail:
  • pad-y / font ∈ [0.50, 0.75] at MD and above
  • pad-x / font ∈ [1.00, 1.50] at canonical (16px), then capped at 32px for XL and 2XL
  • SM is the density / utility tier — intentionally below the Y floor, matching Bootstrap, Polaris, Mantine, Ant, and shadcn convention

Buttons, tabs, and dropdown triggers share the same tokens but obey a slightly different rule above LG — they don't cap X. See buttons for the parallel ratio table.

Content surfaces (alerts, toasts) run a separate, looser band because they hold multi-line prose where vertical headroom matters more than horizontal trim. See alerts & badges.
