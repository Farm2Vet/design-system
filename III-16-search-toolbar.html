III-16 / Components Search toolbar.

The horizontal control strip that sits above a list view. Three slots: an exposed search input, a row of applied-filter chips, and a + Filter trigger that opens a two-step facet picker. Sort lives on the table column header, not here. The applied filters carry no counts — a count per chip would force a per-facet aggregate query before first paint, which the catalogue can't afford as it grows.

01 / Search toolbar — Anatomy

Anatomy.

One flex row. The search slot has a fixed basis on the leading edge. The filters slot grows; chips wrap rather than scroll horizontally. The filter popover hangs off the + Filter trigger and is absolutely positioned below it; it sits above the data region (z-50) and below the modal layer.

Counts on chips are out of contract. Every consumer of this component is server-paginated; a count per facet would force an aggregate query before first paint. If a page seems to need counts, the user is asking a question the toolbar isn't the right place to answer — point them at a stats view instead.

02 / Search toolbar — States

States.

The filter popover has two steps. Step 1 lets the user pick which facet to filter on. Step 2 lets them pick one or more values for that facet, with a search input for long lists and a Clear / Apply footer. Both panels live in the same DOM; the --step-values modifier on .search-toolbar__menu flips which renders.

03 / Search toolbar — Tokens

Tokens.

All component tokens reference semantic tokens — none reach into primitives. Tokens are grouped by slot: row, search, filters, menu, option.
