III-03 / Components Form.

Eight input primitives that share a single shape, scale and state vocabulary — text, textarea, dropdown, date, multi-select, radio, file upload and checkbox. Each resolves through the same --form-* token layer so a tweak to focus colour or radius cascades across every control.

Pair every input with a visible label. Placeholder text disappears the moment the user types and is invisible to most assistive technology — it can carry an example (e.g. Bessie) but never an instruction. Format constraints, ranges, and unit hints belong in helper text below the field. Match the input type to the data — date, email, tel, number — so platforms surface the right keyboard, picker, and validation. Required fields communicate twice: the visible * for sighted readers and aria-required="true" (or the required attribute) for screen readers. Errors live below the field with a specific, actionable message — never a silent red border.

Default to .dropdown on every selector across the product. .select is kept only for edge cases — bulk-edit grids, internal admin tooling, and mobile-native picker flows. See Dropdown · Usage for the exact boundary.

Use a radio group when exactly one option from a small fixed set must be chosen. Pre-select the safest default — leaving the group empty forces a decision before the user knows the cost of each choice. Group every option under a single legend or visible label, and keep the option labels mutually exclusive. Reach for the bordered list variant once labels carry explanation or the set grows past four options; the inline group is for short, single-word choices.

04 / Form — Component tokens

Tokens.

Every form declaration resolves through this layer. Component tokens only ever reference semantic tokens — never primitives — so a focus colour change cascades to inputs, radios, checkboxes and the file-upload dropzone in lockstep.
