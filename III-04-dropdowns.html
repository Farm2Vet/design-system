III-04 / Components Dropdown.

The system's default selector. Reach for .dropdown on every form, filter, and picker across the product — it carries the brand into the most-touched surface in any UI, with primary-tinted option highlights, a checkmark on the active choice, and a 180ms slide-fade-down on open. The native .select remains in the system but is documented strictly as a fallback for the narrow edge cases listed below. If you have to ask which to use, the answer is .dropdown.

01 / Dropdown — Usage

Usage.

Defines the boundary between the system default — .dropdown, used everywhere — and the narrow set of edge cases where the native .select still earns its place.

Default to .dropdown on every product surface a user will actually see. The premium dropdown is the brand's most repeated micro-interaction — opening it should feel like F2V, not like the OS.

  • Intake forms — new patient, new visit, new case.
  • Filters on dashboards, tables, and search results.
  • Settings, profile, and account preferences.
  • Onboarding flows where polish builds trust.
  • Marketing-page forms — contact, demo, sign-up.
  • Anywhere options carry descriptions, icons, or rich content.

Reach for .select only when the native control's constraints actually serve the user better. If the case isn't on the list below, it isn't an edge case.

  • Bulk-edit grids with hundreds of selectors per screen — the JS cost of a custom dropdown per row is real.
  • Internal admin tooling not exposed to customers.
  • Mobile-first flows where OS keyboard or spinner integration is the whole point — date, time, and native pickers.
  • Quick prototypes before the design is locked.
02 / Dropdown — Variants

Variants.

Three variants, each tuned to a distinct user task: making a first-time choice, revising a saved value, or weighing options with explanatory copy.

Placeholder · first-time selection
.is-emptyno prior value
With value · editing existing data
.is-selectedcommitted value
With descriptions · comparing options
.dropdown__option-metalabel + description

Match the variant to the user's task, not to the field. A first-time choice reads differently than revising a saved value, and weighing options with descriptions is a different mental task than picking from a flat list. Don't default to the placeholder variant out of habit — when re-opening a record with an existing value, render with-value; when the user must compare options, use descriptions.

  • Placeholder · first-time selection. New-record forms (new patient, new visit, new case), onboarding intake, filters before the user has narrowed their view. Placeholder copy should mirror the field label as a quiet call to action — "Choose species…" not "Select an option".
  • With value · editing existing data. Edit-record forms (patient details, account settings), persisted filters after a page reload, any form a user has opened to revise — not create. The current value replaces the placeholder; the open menu marks it with a checkmark so the user can re-confirm without scanning.
  • With descriptions · comparing options. Plan / pricing pickers, persona or role selection during onboarding, configuration choices with real consequences (storage tier, region, model size). Reach for this whenever the option's label alone isn't enough for the user to decide. Wrap the secondary line in .dropdown__option-meta; the trigger always shows the label, never the description.
03 / Dropdown — States

States.

Three states shared with every other form control in the system: default (no value yet), with value (the user has committed a choice), and error (validation failed).

Default · before selection
This is .field__helper. Used to guide user.
.is-emptyplaceholder shown
With value · after selection
.is-selectedcommitted value
Error · validation failed
Select a species to continue
.is-errorblocks submission

Apply .is-empty on the trigger to render the placeholder colour. Placeholder copy should phrase the field as a quiet ask, not a generic "Select…". Rendered after the user picks an option, or when re-opening a saved record. Drop the .is-empty class so text colour shifts to --text-default; the matching option in the menu carries .is-selected and shows the checkmark for fast re-confirmation.

04 / Dropdown — Sizes

Sizes.

Five steps along the same scale that drives Type, Button, and Form. Canonical: Large · 22.5px.

SM · 13.33px
MD · 16px
LG · 22.5px · focus
XL · 27.65px
2XL · 33.18px

Pick by surface, not importance. SM lives in dense table headers and admin toolbars; MD is the product default for forms in dialogs and edit panels; LG (canonical) anchors marketing pages, hero intake, and onboarding flows; XL and 2XL are reserved for showcase pickers and display demos. When in doubt on a product surface, default to MD.

Override at the component-token layer, never at the call site. Trigger tokens re-export the --form-* chain so the closed dropdown matches a native .select — change those on :root in the form layer and both follow. Menu, option, and motion tokens are dedicated to this component and own the premium feel; tune --dropdown-anim-easing once and every dropdown across the product opens with a different cadence.
