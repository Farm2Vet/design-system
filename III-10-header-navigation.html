Section 10 / Components Header & Navigation Bar.

The shared page chrome rendered at the top of every design-system page. A two-tier surface: a sticky bar exposing the five top-level Sections of the system, and an expanding panel that lists every Subsection (each a dedicated HTML page) for the three Sections that contain more than one. Resolves through the --global-nav-* pattern back into the system's shared glass, primary, and mono-label tokens.

Architectural note · scope boundaries

The global nav is the system-level surface. It exposes the structure of the design system itself — Primitives, Semantics, Components, Assets, Mockups — and is identical across every documentation page. In-page anchors, on-page tables of contents, breadcrumbs, and product-level navigation belong to other components and must not be merged into this bar.

Architectural note · single source of truth

The bar is rendered by every page from the same markup. The active Subsection is resolved at runtime from <body data-page="…"> by scripts/global-nav.js. Pages do not author per-link active classes; if a page is renamed, only its data-page attribute and the link inside the panel need to change.

01 / Anatomy

Three slots, one row.

The bar (.global-nav__inner) is a three-column grid: brand left, section list centred, version pill right. The expanding panel is positioned against the bar itself — not against any individual section item — so its left and right edges align with the page, regardless of which Section triggered it.

02 / Tier 1 — Section list

Direct links and triggers.

Each tier-1 entry is one of two element types. A direct link (.global-nav__link) points to a single page; a trigger (.global-nav__trigger) is a <button> that toggles its associated panel. Both share the same visual language — a pill-shaped hover background, mono-eyebrow type, and the primary-tinted active state — so that tier-1 reads as a single uniform list even though it mixes the two element types.

Use a direct link when the Section maps to one page. Primitive Tokens and Semantic Tokens each fit on a single page; no expansion is offered for them. Adding an empty panel to a single-page Section dilutes the affordance — the chevron has to mean something.

Use a trigger when the Section is composed of multiple pages. Components, Assets, and Mockups are multi-page Sections. The trigger renders a chevron glyph (.global-nav__chev) that rotates 180° on hover and on .is-open, signalling that the row reveals more content below.

03 / Tier 2 — Panel

A single-purpose surface.

A panel lists every page that belongs to its parent Section. The list is a CSS grid with up to three columns (.global-nav__panel-list) collapsed to one column on narrow viewports. Each entry is a two-part link: a mono index (01, 02, …) followed by the Subsection's title. Subsection numbers match the system's canonical ordering and do not renumber when items are inserted; the index is the page's identity, not its position.

One panel per multi-page Section, never more. Nested panels — a panel that opens a sub-panel — are not supported. If a Section grows beyond a single panel can hold legibly, split it into a new top-level Section with a documented system version bump.

Width, not column count, owns the layout. The panel inner width is capped at the system's shell width (1040px) so its columns line up with page content below. Three columns on desktop, two on tablet, one on mobile. Column count is a function of viewport, not item count.

04 / States

Rest, hover, current.

Three resolved states for tier-1 entries. The active state is applied by the runtime to the Section that contains the current page, so the bar reflects location even when the panel is collapsed.

Inside the panel, the current page is not highlighted. Tier-1 already shows location — the parent Section trigger carries the primary-tinted .is-current state — so reusing the affordance one tier deeper would be redundant. The runtime still applies .is-current to the matched link for downstream tooling, but the panel's CSS does not style it.

Placeholders are dimmed. Subsection links pointing to pages that have not yet been authored carry .is-placeholder. The class lowers the link's contrast so the entry reads as a draft, while still remaining clickable so the structure of the system stays navigable as it's being built out.

05 / Behaviour

Open, dismiss, follow.

Behaviour is split between CSS and the bar's runtime. :focus-within reveal — the keyboard fallback when JavaScript is unavailable — is owned by CSS. Pointer open / close, panel swapping, click-to-toggle, the dim backdrop, and dismiss all live in scripts/global-nav.js; running the runtime is what gives the bar its single-panel-at-a-time guarantee.

Input Outcome
Pointer enters trigger The runtime adds .is-open to the trigger's parent item on mouseenter, which reveals the panel. The panel persists while the pointer is anywhere inside the bar — the gap between triggers, the brand, the version pill, or the panel itself — so users can travel down into the panel without losing it.
Pointer enters another trigger The newly-hovered trigger opens its panel and the previously-open panel closes — including a panel that was opened by click. Only one panel is open at a time; hovering Assets while Components is open swaps cleanly.
Pointer leaves the bar The runtime listens for mouseleave on .global-nav and closes every open panel. Because the panel is a DOM child of the bar, this fires only when the pointer exits both bar and panel — there is no delay-to-close timer.
Trigger receives keyboard focus The panel opens via :focus-within — the one purely-CSS open path, kept as a no-JS fallback. Tab progresses through the panel's links; Shift+Tab returns to the trigger.
Trigger is clicked The runtime toggles .is-open on the parent item and updates aria-expanded. Other open panels are closed first; only one panel is open at a time.
Any panel open The runtime adds .is-visible to a fixed full-viewport .global-nav__backdrop (injected on init), dimming and lightly blurring the page so the panel reads as an elevated surface.
Esc pressed The runtime closes every open panel and resets aria-expanded on each trigger.
Click outside the bar The runtime closes every open panel. Clicks inside the panel are passed through to their target link.
Page scrolls The bar remains pinned at the top via position: sticky. The panel sits absolutely beneath the bar and follows it.
06 / Active page resolution

One attribute, two highlights.

Each page declares its own filename via <body data-page="…">. On DOMContentLoaded the runtime reads that value, finds the matching link inside the panel, and applies .is-current. If the link is inside a panel, the parent Section trigger is also marked .is-current so the bar reflects the active Section even when the panel is collapsed.

The attribute value is the filename without the .html extension. The runtime accepts either form, but pages should normalise to the bare stem. 

<!-- Single line each page must change -->
<body data-page="III-10-header-navigation">

Renaming a page is a two-step edit. Update data-page on the page itself, and update the link inside the matching panel of every page (the panel markup is identical across pages, so a single search-replace across the directory completes the rename).

07 / Accessibility

Roles, labels, focus.

The bar uses native semantics where possible. The outer <header> declares aria-label="Design system navigation"; the section list is a <nav> with its own aria-label; each Section's panel is a role="region" with the Section's name as label.

  • Trigger Rendered as <button type="button">, not <a>. aria-expanded is kept in sync with the panel's open state by the runtime; aria-controls points to the panel's id.
  • Panel Each panel carries role="region" and an aria-label matching its parent Section name. When closed the panel has visibility: hidden and pointer-events: none, removing it from the tab order without changing the DOM.
  • Keyboard Tab progresses through each tier-1 entry in document order. A trigger opens its panel on focus; subsequent Tab steps move through the panel's links. Shift+Tab reverses. Esc closes the open panel and returns focus to the trigger.
  • Visible focus Focus rings inherit the system's default :focus-visible ring from base.css. The bar does not suppress focus styles.
  • Reduced motion The panel's translate-and-fade transition respects the system's prefers-reduced-motion rule from base.css. No bar-specific animation is added beyond the shared opacity transition.
08 / Responsive

Three breakpoints.

The bar collapses in two stages. Below 960px the version pill is dropped and the section list scrolls horizontally; the panel grid drops to two columns. Below 600px the brand label is hidden and the panel grid collapses to a single column.

  • ≥ 960px Full bar: brand, centred section list, version pill. Panel grid renders three columns.
  • 601–959px Version pill hidden; section list right-aligned and horizontally scrollable. Panel grid renders two columns.
  • ≤ 600px Brand label hidden (logo only). Panel grid collapses to a single column.
09 / Usage

Drop in, declare the page.

Every page renders the same nav markup. Three lines of per-page configuration are required: the data-page attribute, the global-nav.css link, and the global-nav.js script. 

<!-- 1. Stylesheet, after shell.css -->
<link rel="stylesheet" href="styles/global-nav.css" />

<!-- 2. Runtime, deferred -->
<script src="scripts/global-nav.js" defer></script>

<!-- 3. Page identity -->
<body data-page="III-02-buttons">

<!-- 4. Bar markup, identical across pages -->
<header class="global-nav" aria-label="Design system navigation">
  <div class="global-nav__inner">
    <!-- brand · sections · version -->
  </div>
</header>

Do not author per-link .is-current classes by hand. The runtime resolves the active page from data-page. Hard-coded active classes drift out of sync with the filename and produce two conflicting highlights.

Do not nest a second navigation surface inside the bar. Breadcrumbs, in-page tables of contents, and contextual sub-navigation are separate components. The bar is reserved for the system-level structure.
