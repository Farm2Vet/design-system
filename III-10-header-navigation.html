III-10 / Components Header & Navigation Bar.

The page chrome rendered at the top of every documentation page. A composition of three floating glass islands — the brand on the left, the sections pill on the right, and a compact version chip in the corner — held by a transparent sticky bar. Multi-page sections expand a right-anchored floating panel that sits just below the sections island with a small breathing gap between them; the sections island alone densifies on open, anchoring the panel to its trigger across the gap as one material event. The bar itself never carries surface — only the islands and the panel do.

Architectural note · scope boundaries

The bar is the system-level surface. It exposes the structure of the design system itself — Primitives, Semantics, Components, Assets, Mockups — and is identical on every documentation page. In-page anchors, on-page tables of contents, breadcrumbs, and product-level navigation belong to other components and are not merged into this bar.

Architectural note · single source of truth

The bar is rendered by every page from the same markup. The active Subsection is resolved at runtime from <body data-page="…"> by scripts/global-nav.js. Pages do not author per-link active classes; renaming a page is a single edit to its data-page value plus the matching link inside the panel.

01 / Composition

Islands.

Three islands compose the bar. The brand drifts to the left edge of the bar's inner padding via margin-right:auto; the sections pill and the version chip cluster on the right. Each island declares its own glass anatomy and its own state machine — none of them shares chrome with its neighbours, and none of them reacts to events on another island.

The bar itself draws nothing. Background, blur, border, and shadow are absent from .global-nav — the bar is a transparent layout container that owns the sticky positioning, the inner padding, and the flex layout. Visual weight is entirely on the islands.

Brand uses the light · light glass tier; sections and chip use the lighter light · clear tier. The contrast pulls the eye to the brand first and lets the sections pill behave as the navigational workhorse without competing for attention.

02 / Panel

Panel.

A panel lists every page in its parent section. When open, it renders as a right-anchored floating card that sits just below the sections island with a small breathing gap between them — enough air that the panel reads as its own surface rather than as a continuation of the pill. The card carries the thick-frost glass and the floating shadow tier; the sections island above it matches the same density so the trigger and the panel still read as one material event, separated only by deliberate air.

Width caps at 760px via max-width on the panel inner. The 3-column grid is preserved for catalog browsing — the system has twelve component pages, three asset pages, and three mockup pages, and a one-column anchored dropdown would cramp the catalog. At narrow viewports the grid drops to two columns, then one, in line with the panel's existing responsive rules.

The panel is right-aligned with the bar's right padding — its right edge sits flush with the version chip's right edge, and the card extends leftward from there. The trigger inside the sections pill is not directly centred over the card; the dropdown anchors to the bar's right column, not to the trigger's centre. This is deliberate: the panel is always-in-the-same-place chrome, not a balloon tied to a moving trigger.

One panel per multi-page section. Nested panels are not supported; if a section grows beyond a single panel, split it into a new top-level section with a documented version bump.

03 / States

States.

Three surface states resolve the bar. At rest every island carries the light-frost variant of its glass tier. When content scrolls under the bar, all three islands step up in density together — light → thick frost, raised → floating shadow, soft → strong rim. When a panel opens, only the sections island densifies; the brand and chip stay at rest because the panel is not their event.

Inside the panel, the current page is not highlighted. The sections island already shows location — the parent section trigger carries the primary-tinted .is-current state — so reusing the affordance one surface deeper would be redundant. The runtime still applies .is-current to the matched link for downstream tooling, but the panel's CSS does not style it.

Placeholders are dimmed. Subsection links pointing to pages that have not yet been authored carry .is-placeholder. The class lowers contrast so the entry reads as a draft while remaining clickable.

04 / Active page resolution

Active page resolution.

Each page declares its own filename via <body data-page="…">. On DOMContentLoaded the runtime reads that value, finds the matching link inside the panel, and applies .is-current. If the link is inside a panel, the parent Section trigger is also marked .is-current so the bar reflects the active Section even when the panel is collapsed.

The attribute value is the filename without the .html extension. The runtime accepts either form, but pages should normalise to the bare stem. 

<!-- Single line each page must change -->
<body data-page="III-10-header-navigation">

Renaming a page is a two-step edit. Update data-page on the page itself, and update the link inside the matching panel of every page (the panel markup is identical across pages, so a single search-replace across the directory completes the rename).

05 / Accessibility

Accessibility.

The bar uses native semantics where possible. The outer <header> declares aria-label="Design system navigation"; the section list is a <nav> with its own aria-label; each Section's panel is a role="region" with the Section's name as label.

  • Trigger Rendered as <button type="button">, not <a>. aria-expanded is kept in sync with the panel's open state by the runtime; aria-controls points to the panel's id.
  • Panel Each panel carries role="region" and an aria-label matching its parent Section name. When closed the panel has visibility: hidden and pointer-events: none, removing it from the tab order without changing the DOM.
  • Chip The version chip is rendered as a static <a> with an explicit aria-label="Design system version". Its href="#" is a placeholder until a versions/changelog page exists; until then the chip serves purely as a typographic anchor at the right edge of the bar.
  • Keyboard Tab progresses through each tier-1 entry in document order. A trigger opens its panel on focus; subsequent Tab steps move through the panel's links. Shift+Tab reverses. Esc closes the open panel and returns focus to the trigger.
  • Visible focus Focus rings inherit the system's default :focus-visible ring from base.css. The bar does not suppress focus styles.
  • Reduced motion The panel's translate-and-fade transition respects the system's prefers-reduced-motion rule from base.css. No bar-specific animation is added beyond the shared opacity transition.
06 / Responsive

Responsive.

The bar collapses in one stage. Below 960px the section list scrolls horizontally and the panel grid drops to two columns. Below 600px the brand label is hidden and the panel grid collapses to a single column.

  • ≥ 960px Full bar: brand island drifts left; sections island and version chip cluster right. Panel inner caps at 760px and right-aligns with the chip's right edge. Panel grid renders three columns.
  • 601–959px Section list scrolls horizontally inside its pill. Panel inner removes the max-width cap and matches the bar's narrow padding. Panel grid renders two columns.
  • ≤ 600px Brand sub-label hidden (logo only). Panel grid collapses to a single column. Chip remains visible.
07 / Usage

Usage.

Every page renders the same nav markup. Three lines of per-page configuration are required: the data-page attribute, the global-nav.css link, and the global-nav.js script. 

<!-- 1. Stylesheet, after shell.css -->
<link rel="stylesheet" href="styles/global-nav.css" />

<!-- 2. Runtime, deferred -->
<script src="scripts/global-nav.js" defer></script>

<!-- 3. Page identity -->
<body data-page="III-02-buttons">

<!-- 4. Bar markup, identical across pages -->
<header class="global-nav" aria-label="Design system navigation">
  <div class="global-nav__inner">
    <!-- brand island · sections island · version chip -->
  </div>
</header>

Do not author per-link .is-current classes by hand. The runtime resolves the active page from data-page. Hard-coded active classes drift out of sync with the filename and produce two conflicting highlights.

Do not nest a second navigation surface inside the bar. Breadcrumbs, in-page tables of contents, and contextual sub-navigation are separate components. The bar is reserved for the system-level structure.
